MVP schema — 4 tables, nothing more
What MVP needs to work end-to-end
Upload xlsx → parse into L0–L3 capability tree → render on canvas
Type a prompt → AI generates diff → Preview Panel → Apply or Cancel
In-session undo/redo (Zustand handles this — no DB needed)
Export PNG, JSON, PPTX
Basic diff log — which prompt changed what (lightweight audit)
What MVP deliberately skips
Multi-user auth and ownership — single user or shared instance is fine for MVP
RAG knowledge base / capability_chunks / pgvector — AI calls Claude without RAG first, add RAG in v2
Cross-session undo (full diff replay) — in-session undo via Zustand is sufficient for MVP
Collaboration / catalog_shares — single consultant per map in MVP
Prompt quality monitoring / prompt_sessions — not needed until AI is live with real users
Soft delete / is_deleted — hard delete is fine at MVP scale, add soft delete in v2
The 4 MVP tables
Table 1 — capability_catalogs — your existing table + industry + status columns added
Table 2 — capabilities — your existing table + updated_at + source columns added
Table 3 — visual_maps — your existing table + version_number + is_active added
Table 4 — diff_history — new, lightweight version. Just enough to power version history sidebar and basic undo awareness
Your 3 original tables stay — just minor column additions. One new table (diff_history, slim version). That is the complete MVP schema. Build everything else only when real users ask for it.

Important design rule for visual_maps in MVP
Every time a user clicks Apply Changes, insert a new row into visual_maps (new version) rather than updating the existing row. Set is_active = true on the new row and false on all previous rows for that catalog. This gives you a free version history without any extra infrastructure.
-- When user applies a diff: -- Step 1: deactivate current layout UPDATE visual_maps SET is_active = false WHERE catalog_id = $catalog_id; -- Step 2: insert new version INSERT INTO visual_maps (catalog_id, name, version_number, is_active, layout_data) VALUES ( $catalog_id, 'Map v' || $next_version, $next_version, true, $new_layout_data );
-- MVP modification: add 2 columns ALTER TABLE visual_maps ADD COLUMN version_number integer DEFAULT 1, ADD COLUMN is_active boolean DEFAULT true;
Table 4 — diff_history (new, slim MVP version)
Why you need this even at MVP
Without diff_history, the version history sidebar in the UI shows visual layout snapshots but cannot tell the user what changed or which prompt caused it.
It also gives you the prompt text back — so if Claude produces a bad result you can see exactly what the user typed.
This is a lightweight version — 7 fields only. No user_id, no model tracking, no retry counts for MVP.

Type
Notes

uuid PK
Auto-generated

→ capability_catalogs.id, cascade delete. Which catalog this diff belongs to.

text NOT NULL
The exact user prompt — "Add risk management capabilities for a retail bank". Critical for debugging and version history display.
diff_payload
jsonb NOT NULL
The full JSON diff array Claude returned — [{action, level, name, parent_id}...]. Enables replaying the change if needed later.
status
text
Default "applied". Set to "cancelled" if user clicked Cancel. Simple enough for MVP — tells you which diffs actually changed the map.
visual_map_id
uuid FK nullable
→ visual_maps.id. Links this diff to the layout snapshot it produced. Null if cancelled. Lets the version history sidebar connect "what changed" to "how it looked".
created_at
timestamptz
Default now(). When the diff was generated and shown in Preview Panel.
Skipped from the full schema for MVP

applied_at — skip. created_at is sufficient for MVP timeline display.
-- Create the slim MVP diff_history table CREATE TABLE diff_history ( id uuid PRIMARY KEY DEFAULT gen_random_uuid(), catalog_id uuid REFERENCES capability_catalogs(id) ON DELETE CASCADE, prompt_text text NOT NULL, diff_payload jsonb NOT NULL, status text DEFAULT 'applied', -- 'applied' | 'cancelled' visual_map_id uuid REFERENCES visual_maps(id) ON DELETE SET NULL, created_at timestamptz DEFAULT now() ); CREATE INDEX ON diff_history(catalog_id); CREATE INDEX ON diff_history(created_at DESC);

What this script does
Modifies your 3 existing tables with the minimum additions
Creates the 1 new table (diff_history, slim version)
Adds the updated_at auto-trigger on capabilities
Adds one useful extra index on capabilities sort_order
Safe to run — uses IF NOT EXISTS and IF NOT COLUMN patterns where possible
-- ═══════════════════════════════════════════ -- MVP SCHEMA — run in Supabase SQL editor -- ═══════════════════════════════════════════ -- ── TABLE 1: capability_catalogs ────────── -- Add industry and status columns ALTER TABLE capability_catalogs ADD COLUMN IF NOT EXISTS industry text, ADD COLUMN IF NOT EXISTS status text DEFAULT 'active'; COMMENT ON COLUMN capability_catalogs.industry IS 'Banking | Healthcare | Retail — sent to AI as RAG filter'; COMMENT ON COLUMN capability_catalogs.status IS 'active | archived — soft archive without deleting'; -- ── TABLE 2: capabilities ───────────────── -- Add source, updated_at, and auto-trigger ALTER TABLE capabilities ADD COLUMN IF NOT EXISTS source text DEFAULT 'xlsx_import', ADD COLUMN IF NOT EXISTS updated_at timestamptz DEFAULT now(); COMMENT ON COLUMN capabilities.source IS 'xlsx_import | ai_generated | manual'; -- Auto-update updated_at on every row change CREATE OR REPLACE FUNCTION update_updated_at_column() RETURNS TRIGGER AS $$ BEGIN NEW.updated_at = now(); RETURN NEW; END; $$ LANGUAGE plpgsql; DROP TRIGGER IF EXISTS capabilities_updated_at ON capabilities; CREATE TRIGGER capabilities_updated_at BEFORE UPDATE ON capabilities FOR EACH ROW EXECUTE FUNCTION update_updated_at_column(); -- Extra index for ordered retrieval (MVP useful) CREATE INDEX IF NOT EXISTS capabilities_catalog_sort_idx ON capabilities(catalog_id, sort_order); -- ── TABLE 3: visual_maps ────────────────── -- Add version_number and is_active ALTER TABLE visual_maps ADD COLUMN IF NOT EXISTS version_number integer DEFAULT 1, ADD COLUMN IF NOT EXISTS is_active boolean DEFAULT true; COMMENT ON COLUMN visual_maps.version_number IS 'Increments per catalog on each Apply — enables version history sidebar'; COMMENT ON COLUMN visual_maps.is_active IS 'Only one row per catalog is true at a time — the current layout'; -- Index for fast "get active layout for this catalog" CREATE INDEX IF NOT EXISTS visual_maps_active_idx ON visual_maps(catalog_id, is_active); -- ── TABLE 4: diff_history (NEW) ─────────── -- Slim MVP version — 7 fields only CREATE TABLE IF NOT EXISTS diff_history ( id uuid PRIMARY KEY DEFAULT gen_random_uuid(), catalog_id uuid REFERENCES capability_catalogs(id) ON DELETE CASCADE, prompt_text text NOT NULL, diff_payload jsonb NOT NULL, status text DEFAULT 'applied', visual_map_id uuid REFERENCES visual_maps(id) ON DELETE SET NULL, created_at timestamptz DEFAULT now() ); COMMENT ON TABLE diff_history IS 'Lightweight audit log — every prompt that reached the AI'; COMMENT ON COLUMN diff_history.status IS 'applied | cancelled'; COMMENT ON COLUMN diff_history.visual_map_id IS 'Links to the visual_maps row created by this diff — null if cancelled'; CREATE INDEX IF NOT EXISTS diff_history_catalog_idx ON diff_history(catalog_id); CREATE INDEX IF NOT EXISTS diff_history_created_idx ON diff_history(created_at DESC); -- ═══════════════════════════════════════════ -- MVP SCHEMA COMPLETE -- 3 modified tables + 1 new table -- Total: 4 tables, ready to build on -- ═══════════════════════════════════════════
