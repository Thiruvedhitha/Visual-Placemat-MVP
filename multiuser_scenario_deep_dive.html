The key question this scenario raises
All three users upload the same SPM template spreadsheet — does the system create 3 separate copies or share one?
Can Alice see Bob's AMEX map? Can Bob see Carol's Takeda map?
When Alice's AI suggests "loyalty program capabilities" using RAG, does it accidentally pull context from Bob's AMEX banking map?
When Carol's Takeda map is approved and ingested into the knowledge base, does it help or contaminate future prompts?
Short answers
3 separate copies are stored — one catalog per client per user. The template is the starting point, not a shared object.
Alice cannot see Bob or Carol's maps — Row Level Security (RLS) in Supabase enforces this at the database level.
RAG is filtered by industry tag — Alice's Retail query never retrieves Banking or Healthcare chunks.
Completed client maps are anonymised before ingestion — "AMEX" becomes "banking-client-uuid-xyz" in the knowledge base.

Complete flow — all 3 users, end to end
Phase 1 — All 3 users upload the same SPM template
1
Alice logs in → uploads SPM_Template.xlsx → selects "Airline"
System creates a new row in capability_catalogs with user_id = alice, industry = "Airline", client_name = "SWA". The spreadsheet parser reads the xlsx and inserts ~40 rows into capabilities, all with catalog_id = cat-swa-uuid-001. Alice sees her SWA map on the canvas.
2
Bob logs in → uploads same SPM_Template.xlsx → selects "Banking"
System creates a completely separate row in capability_catalogs with user_id = bob, industry = "Banking", client_name = "AMEX". Parser inserts ~40 rows into capabilities, all with catalog_id = cat-amex-uuid-002. Bob sees his AMEX map — identical structure to Alice's at this point but a completely isolated dataset.
3
Carol logs in → uploads same SPM_Template.xlsx → selects "Healthcare"
Same pattern — new row in capability_catalogs, user_id = carol, industry = "Healthcare", client_name = "Takeda". ~40 capability rows with catalog_id = cat-takeda-uuid-003. Carol sees her Takeda map.
Phase 2 — Each user types a prompt
4
Alice types: "Add loyalty program capabilities"
HTTP POST to /api/prompt carries: prompt text + Alice's SWA nodes (catalog_id = cat-swa-uuid-001) + industry = "Airline". Backend embeds the prompt → pgvector searches capability_chunks WHERE industry = "Airline" → retrieves top-5 Airline-tagged chunks only. Context Builder assembles payload → Claude returns diff → Diff Normalizer cleans it → Diff Validator checks it → Preview Panel shows Alice the proposed changes.
-- Alice's RAG search is industry-scoped: SELECT content FROM capability_chunks WHERE industry = 'Airline' ORDER BY embedding <=> $alice_prompt_vector LIMIT 5; -- Never touches Banking or Healthcare chunks
5
Bob types: "Add risk management capabilities"
HTTP POST carries: Bob's AMEX nodes (catalog_id = cat-amex-uuid-002) + industry = "Banking". RAG searches WHERE industry = "Banking" only → retrieves banking risk patterns. Claude produces a banking-specific diff. Bob's Preview Panel shows Risk Management, Credit Risk, Operational Risk, Regulatory Compliance — the standard banking hierarchy.
6
Carol types: "Add clinical operations capabilities"
HTTP POST carries: Carol's Takeda nodes + industry = "Healthcare". RAG searches WHERE industry = "Healthcare" only. Claude produces healthcare-specific capabilities. Complete isolation — no cross-contamination between the three users' contexts.
Phase 3 — Apply changes and save
7
Each user clicks Apply Changes
Apply Engine mutates Zustand state → new capability rows inserted into capabilities with the correct catalog_id. A new row is inserted into diff_history with catalog_id scoped to that user's catalog. A new row is inserted into visual_maps for the updated layout. All three operations happen independently, in parallel, with no interference.
8
Ingestion pipeline runs post-approval
Each approved map's new nodes are embedded and added to capability_chunks. Alice's loyalty program nodes go in tagged as industry = "Airline". Bob's risk nodes tagged as industry = "Banking". Carol's clinical ops tagged as industry = "Healthcare". Next time anyone works on a similar client, the RAG knowledge base has real examples to draw from.
How isolation and security are enforced — Supabase RLS
The problem without RLS
Without Row Level Security, any user who knows a catalog_id can query it directly via the Supabase API — even if they do not own it. Alice could read Bob's AMEX map.
The application layer can enforce this but a clever user could bypass it by calling the Supabase REST API directly with someone else's catalog_id.
RLS policies — enforce ownership at the database level
-- Enable RLS on all user-scoped tables ALTER TABLE capability_catalogs ENABLE ROW LEVEL SECURITY; ALTER TABLE capabilities ENABLE ROW LEVEL SECURITY; ALTER TABLE visual_maps ENABLE ROW LEVEL SECURITY; ALTER TABLE diff_history ENABLE ROW LEVEL SECURITY; -- ── capability_catalogs ──────────────────────────────── -- Users can only see their own catalogs CREATE POLICY "owner_select" ON capability_catalogs FOR SELECT USING (user_id = auth.uid()); CREATE POLICY "owner_insert" ON capability_catalogs FOR INSERT WITH CHECK (user_id = auth.uid()); CREATE POLICY "owner_update" ON capability_catalogs FOR UPDATE USING (user_id = auth.uid()); CREATE POLICY "owner_delete" ON capability_catalogs FOR DELETE USING (user_id = auth.uid()); -- ── capabilities ─────────────────────────────────────── -- Users can only see capabilities in their own catalogs CREATE POLICY "owner_capabilities" ON capabilities FOR ALL USING ( catalog_id IN ( SELECT id FROM capability_catalogs WHERE user_id = auth.uid() ) ); -- ── visual_maps ──────────────────────────────────────── CREATE POLICY "owner_visual_maps" ON visual_maps FOR ALL USING ( catalog_id IN ( SELECT id FROM capability_catalogs WHERE user_id = auth.uid() ) ); -- ── diff_history ─────────────────────────────────────── CREATE POLICY "owner_diff_history" ON diff_history FOR ALL USING ( catalog_id IN ( SELECT id FROM capability_catalogs WHERE user_id = auth.uid() ) );
What RLS means in practice for our 3 users
-- Alice queries capabilities — only sees her SWA nodes SELECT * FROM capabilities; -- Supabase automatically adds: -- WHERE catalog_id IN (SELECT id FROM capability_catalogs -- WHERE user_id = 'user-alice-uuid-001') -- Result: only cap-001 through cap-004 (SWA nodes) -- cap-101 to cap-103 (AMEX) are invisible to Alice -- Bob queries catalogs — only sees his AMEX catalog SELECT * FROM capability_catalogs; -- Result: only cat-amex-uuid-002 -- cat-swa-uuid-001 and cat-takeda-uuid-003 are invisible to Bob
These policies apply to every query — SELECT, INSERT, UPDATE, DELETE — whether the request comes from the frontend app, a direct API call, or even the Supabase dashboard if accessed by a non-admin user.
capability_chunks does NOT get RLS — it is a shared knowledge base that all users benefit from. Only catalog ownership tables are isolated.
Collaboration — when Alice wants Bob to review her SWA map
-- catalog_shares enables controlled access INSERT INTO catalog_shares (catalog_id, user_id, role) VALUES ('cat-swa-uuid-001', 'user-bob-uuid-002', 'viewer'); -- Updated RLS policy for capability_catalogs to include shares: CREATE POLICY "owner_or_shared" ON capability_catalogs FOR SELECT USING ( user_id = auth.uid() -- owner OR id IN ( -- shared with this user SELECT catalog_id FROM catalog_shares WHERE user_id = auth.uid() ) );
How RAG works across all 3 clients — isolation without silos
The RAG knowledge base is shared but filtered — not isolated
capability_chunks is a shared table — all industry templates and past client maps go into it. But every query is filtered by industry tag. This is the key design: shared storage, isolated retrieval.
Alice's Airline prompt never retrieves Banking or Healthcare chunks. Bob's Banking prompt never retrieves Airline or Healthcare chunks. The industry tag is the firewall.

What happens if two users work on the same industry client type concurrently
Both users' prompts hit the API simultaneously — no conflict. Each request carries its own catalog_id and industry. They run through the pipeline independently. PostgreSQL handles concurrent inserts to capabilities with row-level locking automatically.
Both users' approved changes get ingested into capability_chunks independently — new rows are appended, no updates to existing rows, no conflict.
The knowledge base gets richer faster when multiple consultants work simultaneously on similar engagements — that is a feature, not a bug.

-- ══════════════════════════════════════════════ -- FULL SCHEMA — all 7 tables -- Wired for multi-user, multi-client scenario -- ══════════════════════════════════════════════ -- Prerequisite: enable pgvector CREATE EXTENSION IF NOT EXISTS vector; -- ── 1. capability_catalogs ───────────────── CREATE TABLE capability_catalogs ( id uuid PRIMARY KEY DEFAULT gen_random_uuid(), user_id uuid REFERENCES auth.users(id) ON DELETE CASCADE, name text NOT NULL, -- "AMEX Capability Map v1" description text, client_name text, -- "AMEX" (display only, not used in RAG) industry text, -- "Banking" — sent to RAG filter status text DEFAULT 'active', -- "active" | "archived" created_at timestamptz DEFAULT now(), updated_at timestamptz DEFAULT now() ); -- ── 2. capabilities ──────────────────────── CREATE TABLE capabilities ( id uuid PRIMARY KEY DEFAULT gen_random_uuid(), catalog_id uuid REFERENCES capability_catalogs(id) ON DELETE CASCADE, parent_id uuid REFERENCES capabilities(id) ON DELETE SET NULL, level smallint NOT NULL CHECK (level BETWEEN 0 AND 3), name text NOT NULL, description text, sort_order integer DEFAULT 0, source text DEFAULT 'xlsx_import', -- "xlsx_import" | "ai_generated" is_deleted boolean DEFAULT false, -- soft delete for undo support created_at timestamptz DEFAULT now(), updated_at timestamptz DEFAULT now() ); CREATE INDEX ON capabilities(catalog_id); CREATE INDEX ON capabilities(parent_id); CREATE INDEX ON capabilities(catalog_id, level); CREATE INDEX ON capabilities(catalog_id, sort_order); -- Auto-update updated_at CREATE OR REPLACE FUNCTION update_updated_at_column() RETURNS TRIGGER AS $$ BEGIN NEW.updated_at = now(); RETURN NEW; END; $$ LANGUAGE plpgsql; CREATE TRIGGER capabilities_updated_at BEFORE UPDATE ON capabilities FOR EACH ROW EXECUTE FUNCTION update_updated_at_column(); -- ── 3. visual_maps ───────────────────────── CREATE TABLE visual_maps ( id uuid PRIMARY KEY DEFAULT gen_random_uuid(), catalog_id uuid REFERENCES capability_catalogs(id) ON DELETE CASCADE, name text NOT NULL, version_number integer DEFAULT 1, layout_data jsonb, -- React Flow node/edge positions is_active boolean DEFAULT true, thumbnail_url text, created_at timestamptz DEFAULT now(), updated_at timestamptz DEFAULT now() ); CREATE INDEX ON visual_maps(catalog_id, is_active); -- ── 4. diff_history ──────────────────────── CREATE TABLE diff_history ( id uuid PRIMARY KEY DEFAULT gen_random_uuid(), catalog_id uuid REFERENCES capability_catalogs(id) ON DELETE CASCADE, applied_by uuid REFERENCES auth.users(id) ON DELETE SET NULL, prompt_text text NOT NULL, diff_payload jsonb NOT NULL, status text DEFAULT 'applied', -- "applied"|"cancelled"|"rolled_back" model_used text, -- "claude-sonnet-4"|"phi-3"|"llama-3-8b" visual_map_id uuid REFERENCES visual_maps(id) ON DELETE SET NULL, created_at timestamptz DEFAULT now(), applied_at timestamptz ); CREATE INDEX ON diff_history(catalog_id); CREATE INDEX ON diff_history(applied_by); -- ── 5. capability_chunks (RAG) ───────────── CREATE TABLE capability_chunks ( id uuid PRIMARY KEY DEFAULT gen_random_uuid(), label text NOT NULL, level text NOT NULL, industry text NOT NULL, -- the isolation filter content text NOT NULL, -- readable text sent to Claude embedding vector(1536), -- pgvector column source text NOT NULL, -- "template" | "client_map" source_catalog_id uuid REFERENCES capability_catalogs(id) ON DELETE SET NULL, created_at timestamptz DEFAULT now() ); -- IVFFlat index for fast cosine similarity search CREATE INDEX ON capability_chunks USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100); CREATE INDEX ON capability_chunks(industry); -- ── 6. prompt_sessions ───────────────────── CREATE TABLE prompt_sessions ( id uuid PRIMARY KEY DEFAULT gen_random_uuid(), catalog_id uuid REFERENCES capability_catalogs(id) ON DELETE CASCADE, user_id uuid REFERENCES auth.users(id) ON DELETE SET NULL, prompt text NOT NULL, model_used text, retry_count smallint DEFAULT 0, validation_error text, latency_ms integer, created_at timestamptz DEFAULT now() ); CREATE INDEX ON prompt_sessions(catalog_id); -- ── 7. catalog_shares ────────────────────── CREATE TABLE catalog_shares ( id uuid PRIMARY KEY DEFAULT gen_random_uuid(), catalog_id uuid REFERENCES capability_catalogs(id) ON DELETE CASCADE, user_id uuid REFERENCES auth.users(id) ON DELETE CASCADE, role text DEFAULT 'viewer', -- "viewer"|"editor"|"owner" created_at timestamptz DEFAULT now(), UNIQUE(catalog_id, user_id) ); -- ── RLS POLICIES ─────────────────────────── ALTER TABLE capability_catalogs ENABLE ROW LEVEL SECURITY; ALTER TABLE capabilities ENABLE ROW LEVEL SECURITY; ALTER TABLE visual_maps ENABLE ROW LEVEL SECURITY; ALTER TABLE diff_history ENABLE ROW LEVEL SECURITY; ALTER TABLE prompt_sessions ENABLE ROW LEVEL SECURITY; -- capability_catalogs: owner or shared CREATE POLICY "access_catalogs" ON capability_catalogs FOR ALL USING ( user_id = auth.uid() OR id IN (SELECT catalog_id FROM catalog_shares WHERE user_id = auth.uid()) ); -- capabilities: scoped via catalog ownership CREATE POLICY "access_capabilities" ON capabilities FOR ALL USING ( catalog_id IN ( SELECT id FROM capability_catalogs WHERE user_id = auth.uid() OR id IN (SELECT catalog_id FROM catalog_shares WHERE user_id = auth.uid()) ) ); -- visual_maps: same as capabilities CREATE POLICY "access_visual_maps" ON visual_maps FOR ALL USING ( catalog_id IN ( SELECT id FROM capability_catalogs WHERE user_id = auth.uid() OR id IN (SELECT catalog_id FROM catalog_shares WHERE user_id = auth.uid()) ) ); -- diff_history: same scoping CREATE POLICY "access_diff_history" ON diff_history FOR ALL USING ( catalog_id IN ( SELECT id FROM capability_catalogs WHERE user_id = auth.uid() ) ); -- prompt_sessions: user sees only their own CREATE POLICY "access_prompt_sessions" ON prompt_sessions FOR ALL USING (user_id = auth.uid()); -- capability_chunks: NO RLS — shared knowledge base -- All users benefit from all industry chunks -- Client identity is anonymised at ingestion time
